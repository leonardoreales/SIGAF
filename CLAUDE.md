## Architecture

### Monorepo layout
```
shared/                  → Zod schemas shared between backend and frontend
Backend/api/             → Express REST API (TypeScript)
Backend/migration/       → SQL DDL files + Python Excel importers
Backend/n8n-workflows/   → Automation JSON workflows (ingesta + post-firma)
Frontend/web/            → React SPA (TypeScript + Vite)
assets/                  → Project logos and static assets
data/                    → Excel master files (inventory, responsible parties)
```

### Backend: Clean Architecture layers

`Backend/api/src/` is organized strictly by layer — **never mix them**:

| Layer | Path | Responsibility |
|---|---|---|
| Interfaces | `interfaces/` | Express routers — HTTP only, no business logic |
| Application | `application/` | Use cases — one folder per domain |
| Domain | `domain/` | Core types and event definitions |
| Infrastructure | `infrastructure/db/` | Drizzle ORM repositories |
| Infrastructure | `infrastructure/sse/` | SSE broadcast manager |
| Shared | `shared/middleware/` | authenticate.ts, authorize.ts, error.ts |

**Data flow:** Router → Use case → Repository → Drizzle → PostgreSQL

### Frontend: State and data

- **React Query** handles all server state (assets, transfers, users, etc.). Cache stale time: 30s.
- **AuthContext** stores JWT (`sigaf_token`) and user object (`sigaf_user`) in localStorage. Auto-logout on 401.
- **ThemeContext** controls dark/light mode via Tailwind `class` strategy.
- **useSyncEvents** opens an SSE connection to `/sync/events` and publishes to `activityBus` for live notifications.
- [Frontend/web/src/lib/api.ts](Frontend/web/src/lib/api.ts) is the single HTTP client — all API calls go through typed namespaces (`apiAssets`, `apiTransfers`, etc.).

### RBAC

Four roles defined in `shared/src/index.ts` → `RoleSchema`:
- `ADMIN` — full access
- `ACTIVOS_FIJOS` — assets, transfers, maintenance, writeoffs
- `COMPRAS` — recepciones workflow only
- `VIEWER` — read-only

Role enforcement: `authorize.ts` middleware on the backend; `ProtectedRoute.tsx` + `useRole.ts` on the frontend.

### Database

PostgreSQL with Drizzle ORM. Schema defined in [Backend/api/src/infrastructure/db/schema.ts](Backend/api/src/infrastructure/db/schema.ts) — this is the source of truth for table structure.

Key domain tables: `assets`, `transfers`, `transfer_requests`, `transfer_request_items`, `writeoff_acts`, `writeoff_items`, `maintenance_schedules`, `maintenance_executions`, `maintenance_supports`, `system_users`.

Catalog (reference) tables: `catalog_buildings`, `catalog_areas`, `catalog_asset_types`, `catalog_people`, `catalog_cities`.

Maintenance hierarchy: `maintenance_schedules` (plan) → `maintenance_executions` (execution record) → `maintenance_supports` (file attachments). Executions are validated by a `system_users` reference (`validatedBy`).

`transfer_requests.formData` is a JSONB blob storing the parsed DOCX payload (form F-AF-039) plus `signatureWorkflowResult` written back by n8n after signing. It is the handshake envelope between the backend and n8n — check it before assuming fields are in the relational columns.

Asset plate format: `[CityCode]-[TypeCode]-[Sequence]` (e.g. `M-70-0001`). Auto-generated by `generate_plates_2026.sql`.

### n8n integration

Three workflows in `Backend/n8n-workflows/`:
1. **Ingesta** — polls Gmail every minute, parses DOCX attachments (form F-AF-039), calls `POST /transfers/requests`
2. **Firma** — triggered by `POST /transfers/requests/:id/sign`; inserts the signature image into Google Doc, generates PDF, replies to the original Gmail thread
3. **Post-firma** — n8n calls back `POST /transfers/requests/firma-completada` when done → updates status + SSE broadcast

The backend notifies n8n via `POST /sync/notify` and broadcasts real-time updates to the frontend via SSE at `GET /sync/events`.

#### Transfer request status lifecycle

```
RECIBIDA → PENDIENTE_GESTION_ACTIVOS_FIJOS → REVISION → APROBADA
  → FIRMA_SOLICITADA → FIRMA_EN_PROCESO → PDF_GENERADO
  → RESPUESTA_ENVIANDO → FIRMADA → RESPUESTA_ENVIADA
```

- **SIGNABLE** statuses (can trigger signing): `PENDIENTE_GESTION_ACTIVOS_FIJOS`, `REVISION`, `APROBADA`
- **IN_PROGRESS** statuses (signing already underway — idempotent no-op): `FIRMA_SOLICITADA`, `FIRMA_EN_PROCESO`, `PDF_GENERADO`, `RESPUESTA_ENVIANDO`
- `FIRMADA` / `RESPUESTA_ENVIADA` → already done, return early without error

### Shared Zod schemas

[shared/src/index.ts](shared/src/index.ts) exports all enums and validation schemas used by both backend (input validation) and frontend (form types). When adding a new field or status value, update this file first.

### Environment variables

**Backend** (`Backend/api/.env`):
`DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `PORT` (3000), `GOOGLE_CLIENT_ID`, `JWT_SECRET`, `JWT_EXPIRES_IN` (8h), `FRONTEND_URL`

Signing-specific: `N8N_SIGN_WEBHOOK` (required — n8n webhook URL), `N8N_SIGN_HMAC_SECRET` (optional HMAC for `x-sigaf-signature` header), `N8N_SIGN_REQUEST_SECRET` (sent as `x-sigaf-sign-request-secret`), `SIGAF_SIGNATURE_IMAGE_FILE_ID` (Google Drive file ID of the signature image), `SYNC_SECRET` (fallback secret used if `N8N_SIGN_REQUEST_SECRET` is absent)

**Frontend** (`Frontend/web/.env`):
`VITE_API_URL`, `VITE_GOOGLE_CLIENT_ID`

### Authentication flow

1. Frontend: Google OAuth → gets `id_token` via `@react-oauth/google`
2. Frontend: `POST /auth/google` with `id_token`
3. Backend: `googleVerifier.ts` validates token → issues JWT
4. All subsequent requests: `Authorization: Bearer <jwt>` header
5. `authenticate.ts` middleware decodes JWT and attaches user to `req.user`
